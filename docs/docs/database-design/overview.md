# 数据库设计总览

这部分文档描述 `panda pt site` 的数据库设计思路，目标是让重写工作在建表前就明确：

- 哪些是核心业务表
- 哪些是关系表
- 哪些是日志表
- 哪些字段必须可追溯
- 哪些查询路径必须优先优化

## 设计目标

- 支撑 PT 站核心业务闭环
- 保证用户、种子、做种和运营数据可回溯
- 让高频查询有稳定索引
- 让历史记录和日志可审计
- 让配置和规则可动态调整

## 设计原则

### 1. 业务优先

先围绕业务对象建模，再决定字段和索引，不从页面反推表结构。

### 2. 主体与日志分离

所有关键行为都应该有主对象和日志对象：

- 主对象表示当前状态
- 日志对象表示历史变化

### 3. 状态显式化

审核状态、发布状态、违规状态、完成状态都要明确，不允许混在一个字段里含糊表达。

### 4. 高查询路径优先

高频查询对象优先优化：

- 用户主页
- 种子列表
- tracker 相关状态
- 运营统计
- 管理后台列表

### 5. 可扩展

设计时要预留：

- 业务扩展字段
- 配置扩展字段
- 日志扩展字段
- 插件扩展字段

## 逻辑分层

### 1. 核心业务层

承载用户、种子、Peer、Snatch、考核、H&R、奖励等主业务对象。

### 2. 运营治理层

承载签到、公告、通知、勋章、规则、风控、插件等治理对象。

### 3. 系统配置层

承载站点开关、基础字典、网络规则、搜索配置和后台参数。

### 4. 审计与统计层

承载日志、统计快照、历史轨迹和异常记录。
