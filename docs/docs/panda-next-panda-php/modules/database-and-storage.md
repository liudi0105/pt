# 数据库与存储

## 模块目标

这个模块负责承载站点所有业务数据、运行数据和附件数据。

重写时必须先把“数据怎么活下来”设计清楚，功能层才有稳定基础。

## 存储对象

- 用户主数据
- 种子与做种数据
- 评论、消息、奖励等互动数据
- 签到、考核、H&R 等运营数据
- 配置、风控和后台字典
- 附件、截图、日志、备份文件

## 存储分层

### 1. 主业务数据

这部分数据要求强一致、可追踪、可恢复。

典型内容：

- 用户
- 种子
- Peer
- Snatch
- 考核
- H&R

### 2. 运行态数据

这部分数据偏短周期，适合缓存或队列承载。

典型内容：

- 会话
- 验证状态
- 任务状态
- 索引同步状态
- 临时统计

### 3. 文件与附件

这部分数据体积较大，恢复成本高。

典型内容：

- 种子附件
- 图片和截图
- 备份文件
- 站点资源文件

## 设计要求

### 1. 一致性

关键业务数据必须可回溯，不允许只有最终结果没有过程。

### 2. 可恢复性

备份不是附属功能，而是站点基础能力。

必须考虑：

- 数据库备份
- 文件备份
- 恢复流程
- 备份验证

### 3. 可扩展性

随着站点规模增长，数据模型和存储结构要能支持：

- 更高并发
- 更大附件量
- 更多统计数据
- 更多历史记录

### 4. 可观测性

存储系统要能被监控。

需要关注：

- 磁盘空间
- 备份状态
- 索引状态
- 队列堆积
- 错误日志

## 关键规则

- 主业务数据与运行态数据分开理解
- 附件和日志要考虑生命周期
- 备份必须定期验证
- 搜索索引不是数据源，主库才是数据源

## 依赖模块

- 所有业务模块
- 后台治理
- 搜索与索引
- 运维任务

## 验收标准

- 主要业务对象都能落库并恢复
- 附件和备份有明确归档方式
- 运行状态不会污染主业务数据
- 监控能发现存储和备份异常
