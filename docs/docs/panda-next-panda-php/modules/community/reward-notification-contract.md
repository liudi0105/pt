# 奖励与通知契约

## 1. 概述

定义站内各业务事件触发奖励发放后，如何同步产生并触达用户通知的联动契约。确保用户能及时获知自己的奖励结果和状态变更。

## 2. 功能说明

### 2.1 通知触发事件

以下业务事件会触发系统通知的生成和送达：

#### 2.1.1 评论通知

- **事件**：用户对种子/求种/邀约发表评论。
- **触发条件**：目标资源所有者设置了 `commentpm='yes'`。
- **动作**：系统发送站内信通知该用户"有新的评论"。
- **通知内容**：评论摘要、资源标题、评论作者。

#### 2.1.2 H&R 状态变更通知

- **事件**：H&R 记录状态变为 `REACHED`（达标）或 `UNREACHED`（未达标）。
- **触发条件**：用户 `notifs` 字段开启了 `hr_reached` 通知偏好。
- **动作**：系统发送站内信通知用户 H&R 判定结果。
- **通知内容**：种子名称、当前状态、做种时间和分享率等详情。

#### 2.1.3 考核结果通知

- **事件**：考核过期结算（定时任务 `exam:checkout`）。
- **触发条件**：考核到期自动结算。
- **动作**：
  - 考核通过：发送祝贺站内信。
  - 考核失败：发送失败通知，并附带账号被禁用的原因。
- **通知内容**：考核名称、各指标完成情况、最终结果。

#### 2.1.4 认领结算通知

- **事件**：认领月度结算（定时任务）。
- **触发条件**：认领达到结算周期。
- **动作**：发送详细的月度结算站内信。
- **通知内容**：当月做种时间、上传量、达标状态、奖励/扣罚结果。

#### 2.1.5 用户治理通知

- **事件**：管理员对用户执行封禁/解封/警告等治理动作。
- **触发条件**：治理操作执行完毕。
- **动作**：系统发送治理原因通知给相应用户。
- **通知内容**：治理类型、原因、操作时间。

### 2.2 通知聚合接口

`GET /api/notifications` 将多个来源的通知数据聚合成统一响应：

| 来源 | 数据内容 |
|------|----------|
| 签到 | 当日是否已签到 |
| 新闻 | 自 `last_home` 以来新增新闻数 |
| 站内信 | 未读站内信数量 |
| 投票 | 未投票的投票数量 |

### 2.3 通知偏好管理

- `User::acceptNotification($name)` 方法读取用户 `notifs` 字段 JSON 配置，判断是否接收指定类型的通知。
- 用户可在个人设置中分别开启/关闭以下通知类型：`topic_reply`、`hr_reached`。
- 部分通知（如考核结果、认领结算）为强制性通知，不受用户偏好影响。

## 3. 操作入口

- 通知中心：`GET /api/notifications`
- 站内信列表：`/api/messages`
- 用户通知偏好设置页
- API：`GET /api/notifications`

## 4. 使用说明

1. 通知聚合接口返回的是实时数据，不依赖缓存。
2. 评论通知仅当目标所有者开启 `commentpm` 时发送，不影响评论本身的发布。
3. H&R 通知受用户偏好控制，默认发送。
4. 考核和认领结算通知为系统强制发送，用户无法关闭。
5. 用户治理通知包含封禁/解封原因，用于用户申诉参考。

## 5. 配置参考

通知行为由用户 `notifs` 字段和内建业务逻辑共同控制，无独立系统配置项。
