# 主业务数据存储

## 1. 概述

承载站点核心业务对象（用户、种子、评论、消息、新闻、投票、奖励、考核、H&R、请求、标签等）的持久化存储，是所有业务规则和前后台查询的最终数据源。

## 2. 功能说明

### 2.1 核心业务表

| 表 | 用途 | 关键关系 |
|----|------|----------|
| `users` | 用户主档案 | 与所有业务表关联 |
| `torrents` | 种子信息 | 关联 `users`(发布者)、`comments`、`peers`、`snatches` |
| `comments` | 评论内容 | 关联 `users`(评论者)、`torrents` |
| `messages` | 站内信 | 关联 `users`(发送/接收) |
| `news` | 站点公告 | 关联 `users`(发布者) |
| `polls` | 投票调查 | 关联 `users`(投票者) |
| `rewards` | 奖励记录 | 关联 `users`(获奖者)、运营活动 |
| `medals` | 勋章系统 | 关联 `users`(持有者) |
| `exams` | 考核体系 | 关联 `users`(参与人)、`torrents` |
| `hit_and_runs` | H&R 记录 | 关联 `users`、`torrents`、`snatches` |
| `requests` | 求种记录 | 关联 `users`(发起者)、`torrents`(应求) |
| `claims` | 应求记录 | 关联 `users`(响应者)、`requests` |
| `tags` | 种子标签 | 关联 `torrents`(多对多) |

### 2.2 核心特性
- **强一致性**：用户额度、种子状态等关键业务事实要求数据强一致。
- **关联完整性**：通过外键和业务逻辑保证跨表引用完整性。
- **审核可追溯**：审核、处罚、奖励等操作不仅保留最终结果，还需记录操作过程。
- **主数据唯一性**：搜索索引和缓存不可替代主业务数据源。

### 2.3 数据生命周期
- **创建**：业务模块通过 Repository 或 Eloquent Model 写入主表。
- **更新**：主要字段变更通过 Service 层处理，同步更新关联表。
- **删除/下架**：部分业务对象支持软删除（如种子可标记为下架而非物理删除）。
- **归档**：历史数据根据策略归档至对应日志表（如 `torrent_operation_logs`）。

### 2.4 业务影响
- 评论删除影响前台展示，评论数据本身保留。
- 种子删除需联动处理关联的评论、感谢、书签、Peer 等。
- 用户删除触发级联清理（通过 `UserRepository::destroy()`）。

## 3. 操作入口

- 数据写入：各业务 Repository / Service
- 数据查询：前台 API + 后台 Filament Resource
- 数据管理：Filament 后台各管理页面

## 4. 使用说明

1. 所有新业务对象需在主库建立对应表，遵循现有命名规范。
2. 关联表变更需注意事务一致性，避免产生孤立记录。
3. 核心对象的关键历史变化必须记录到对应审计表。
4. 搜索类功能和索引构建不可依赖主库直接扫描，应使用专用搜索引擎或缓存。

## 5. 配置参考（可选）

- 数据库连接配置在 `config/database.php`
- 部分表支持软删除，通过 `deleted_at` 字段控制
