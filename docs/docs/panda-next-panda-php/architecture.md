# 系统架构

`panda-php` 的架构核心不是页面，而是围绕 PT 站点运行周期构建的“行为闭环”。

用户从注册、登录、浏览种子、下载、做种、签到、完成考核，到被奖励或处罚，整个过程都被系统持续追踪。

## 总体分层

### 1. 接入层

- Web 页面请求
- JSON API 请求
- Tracker 请求
- OAuth 授权请求
- 第三方自动化接入请求

### 2. 应用层

- 请求编排负责把外部输入转成业务动作
- 查询编排负责把业务条件转成可用结果
- 异步处理负责延后执行耗时动作
- 定时处理负责周期性清理、结算和同步
- 事件处理负责在状态变化后联动其他模块

### 3. 领域层

- User、Torrent、Peer、Snatch、Exam、HitAndRun、Medal、Claim、Reward 等模型
- 对应的策略、政策、状态枚举和统计逻辑

### 4. 基础设施层

- MySQL 作为主存储
- Redis 作为缓存、队列或临时状态存储
- Elasticsearch / MeiliSearch 作为搜索索引
- FTP / SFTP / Google Drive 作为外部文件或备份存储

## 核心闭环

### 1. 用户闭环

- 用户注册、登录、访问站点
- 系统识别等级、权限、封禁和偏好
- 用户在站内产生浏览、评论、收藏、签到、下载和做种行为
- 系统把行为沉淀为日志、积分、统计和处罚依据

### 2. 种子闭环

- 发布者上传种子
- 管理员审核和配置可见性
- 用户下载后通过 tracker 上报做种状态
- 系统更新 Snatch、Peer、完成记录和搜索索引

### 3. 运营闭环

- 站点通过签到、考核、H&R、勋章和奖励驱动活跃度
- 后台通过设置、字典和风控规则控制站点秩序
- 定时任务和异步任务负责清理、同步和结算

## 认证体系

- `nexus-web`：面向 Web 登录态
- `api`：面向接口认证
- `passport`：面向 OAuth 授权
- `sanctum`：作为轻量 API / SPA 能力补充
- `admin`：限制后台管理接口
- `user`：限制普通用户 API

## 语言和平台控制

- `locale` 中间件负责多语言
- `Platform` 中间件通常用于站点端能力或访问平台判断
- `BootNexus` 和 `NexusAuth` 用于初始化站点运行上下文

## 关键设计特征

- 强业务域分层，控制器和仓储边界清晰
- PT 协议层独立于普通 Web 业务
- 后台管理和前台使用同一套模型与权限体系
- 搜索、备份、插件、通知都通过独立基础设施承载

## 重写原则

- 不把页面当模块，把用户行为链路当模块
- 不把数据库表当需求，把业务状态当需求
- 不把后台资源当设计，把运营动作和控制规则当设计
- 不把外部接入当附属能力，把 tracker 和搜索当核心系统的一部分
